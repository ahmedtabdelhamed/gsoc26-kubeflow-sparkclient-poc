# Kubeflow SparkClient Batch Job Submission - GSoC 2026 POC

## 🎯 Project Overview

This Proof-of-Concept demonstrates the foundational work for extending the Kubeflow SparkClient to support **batch job submission** using Apache Spark Operator on Kubernetes. This project is proposed for **Google Summer of Code 2026** with the Kubeflow community.

Currently, the Kubeflow SDK's SparkClient only supports **interactive Spark Connect sessions** for exploratory workloads. This POC lays the groundwork for adding production-ready batch job capabilities using `SparkApplication` CRDs.

## 🔑 Critical Context: Issue #2818

**Why This Architecture?**

This POC directly addresses [Kubeflow SDK Issue #2818](https://github.com/kubeflow/spark/issues/2818), which identifies a critical technical debt in the current SDK:

> **Current Problem:** The SDK constructs Spark Custom Resources (CRs) using raw dictionaries, leading to:
> - ❌ No type safety or IDE autocomplete
> - ❌ Error-prone manual field construction
> - ❌ No validation until runtime (after Kubernetes submission)
> - ❌ Difficult to maintain as CRDs evolve

> **Required Solution:** Move to a **structured, model-based approach** using generated Python models from CRD schemas.

### Alignment with Kubeflow Ecosystem

This POC follows the **exact pattern** used by Kubeflow TrainerClient:

1. **TrainerClient Pattern** (Established Best Practice):
   ```python
   from kubeflow_trainer_api import models
   
   # Type-safe model construction
   train_job = models.TrainerV1alpha1TrainJob(
       apiVersion="trainer.kubeflow.org/v1alpha1",
       kind="TrainJob",
       metadata=models.ObjectMeta(name="job-1"),
       spec=models.TrainerV1alpha1TrainJobSpec(...)
   )
   
   # Convert to dict for Kubernetes API
   body = train_job.to_dict()
   ```

2. **SparkClient Pattern** (This POC):
   ```python
   from kubeflow_spark_api import models  # ← Generated package (Phase 0)
   
   # Type-safe model construction
   spark_app = models.SparkApplication(
       apiVersion="sparkoperator.k8s.io/v1beta2",
       kind="SparkApplication",
       metadata=models.ObjectMeta(name="pi-job"),
       spec=models.SparkApplicationSpec(...)
   )
   
   # Convert to dict for Kubernetes API
   body = spark_app.to_dict()
   ```

## 🏗️ Architecture Design

### Three-Layer Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  CLIENT LAYER (User-Facing API)                             │
│  kubeflow.spark.api.SparkClient                             │
│  - submit_job(name, image, main_file, ...)                  │
│  - get_job(name) → SparkJobStatus                           │
│  - list_jobs(status=None) → List[SparkJobStatus]            │
│  - wait_for_job_completion(name, timeout)                   │
│  - delete_job(name)                                          │
└─────────────────────────────────────────────────────────────┘
                           ↓ delegates to
┌─────────────────────────────────────────────────────────────┐
│  BACKEND LAYER (Kubernetes Implementation)                  │
│  kubeflow.spark.backends.kubernetes.KubernetesBackend       │
│  - Uses generated models from kubeflow_spark_api            │
│  - Converts models to dicts via .to_dict()                  │
│  - Submits to Kubernetes CustomObjectsApi                   │
│  - Parses responses with models.*.from_dict()               │
└─────────────────────────────────────────────────────────────┘
                           ↓ uses
┌─────────────────────────────────────────────────────────────┐
│  MODEL LAYER (Generated from CRDs)                          │
│  kubeflow_spark_api.models                                  │
│  - SparkApplication (v1beta2)                               │
│  - SparkApplicationSpec, DriverSpec, ExecutorSpec           │
│  - Type-safe Python classes with validation                 │
│  - Generated via OpenAPI Generator                          │
└─────────────────────────────────────────────────────────────┘
```

### Key Design Patterns

1. **Backend Delegation:**
   - Client layer validates inputs and provides user-friendly API
   - Backend layer handles Kubernetes complexity

2. **Progressive Complexity (3 Levels):**
   - **Level 1:** Simple defaults - `submit_job(name, image, main_file)`
   - **Level 2:** Common parameters - Add `spark_version`, `driver_cores`, `executor_instances`
   - **Level 3:** Full control - Use `Driver`/`Executor` objects with complete configuration

3. **Early Validation:**
   - Type hints and Pydantic models validate at client layer
   - Catch errors before Kubernetes submission

4. **Model-Based CRD Construction:**
   - Replace dictionary manipulation with type-safe models
   - IDE autocomplete and compile-time type checking
   - Schema-driven validation

## 🚀 Phase 0: CRD Schema Extraction (Current POC)

### What This POC Demonstrates

The `fetch_crd_schema.py` script shows understanding of the model generation pipeline:

```python
# 1. Fetch SparkApplication CRD from upstream
CRD_URL = "https://raw.githubusercontent.com/kubeflow/spark-operator/master/config/crd/bases/sparkoperator.k8s.io_sparkapplications.yaml"

# 2. Extract OpenAPI V3 schema
open_api_schema = crd_dict['spec']['versions'][0]['schema']['openAPIV3Schema']

# 3. Save schema for model generation
json.dump(open_api_schema, f, indent=2)
```

### Running Phase 0

```bash
# Extract OpenAPI schema from SparkApplication CRD
python fetch_crd_schema.py

# Output: api_generator/spark_app_openapi.json
```

## 🎓 Why This Approach?

### Benefits of Model-Based Architecture

| Aspect | Dictionary Approach (Current) | Model-Based Approach (This POC) |
|--------|-------------------------------|----------------------------------|
| **Type Safety** | ❌ No type checking | ✅ Full type hints and validation |
| **IDE Support** | ❌ No autocomplete | ✅ Full autocomplete and docs |
| **Error Detection** | ❌ Runtime (after K8s submission) | ✅ Compile-time + early validation |
| **Maintainability** | ❌ Manual field tracking | ✅ Schema-driven, auto-updated |
| **Testing** | ❌ Mock entire dicts | ✅ Mock type-safe objects |
| **Documentation** | ❌ Manual docs drift | ✅ Generated from schema |

### Alignment with Kubeflow Standards

- **TrainerClient:** Uses `kubeflow_trainer_api` models ✅
- **KatibClient:** Uses `kubeflow_katib_api` models ✅
- **SparkClient:** Will use `kubeflow_spark_api` models ✅
